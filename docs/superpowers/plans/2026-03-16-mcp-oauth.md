# MCP OAuth 2.1 Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add MCP-spec-compliant OAuth 2.1 (Authorization Code + PKCE) so ChatGPT and other clients can authenticate via the standard OAuth discovery flow.

**Architecture:** OAuthDriver interface with a built-in driver. OAuth flow issues standard McpTokens via existing TokenService. Discovery endpoints at `.well-known/`, OAuth endpoints at `/mcp/oauth/`, consent screen in Statamic CP.

**Tech Stack:** PHP 8.3, Laravel 12, Statamic v6, Vue 3, PHPStan Level 8, Pest 4

**Spec:** `docs/superpowers/specs/2026-03-16-mcp-oauth-design.md`

### Task 3: Create OAuthException

**Files:**

- [ ] **Step 1: Implement**

```php
<?php

declare(strict_types=1);

namespace Cboxdk\StatamicMcp\OAuth\Exceptions;

class OAuthException extends \RuntimeException
{
    public function __construct(
        public readonly string $errorCode,
        string $description,
        int $httpStatus = 400,
    ) {
        parent::__construct($description, $httpStatus);
    }

    /**
     * @return array{error: string, error_description: string}
     */
    public function toOAuthResponse(): array
    {
        return [
            'error' => $this->errorCode,
            'error_description' => $this->getMessage(),
        ];
    }
}
```

## Chunk 2: BuiltInOAuthDriver

### Task 5: Create BuiltInOAuthDriver with tests

**Files:**
- Create: `src/OAuth/Drivers/BuiltInOAuthDriver.php`
- Create: `tests/Feature/OAuth/BuiltInOAuthDriverTest.php`

This is the core implementation. Key behaviors:

**Client storage** (file-based for now, same dir as token file store):
- Clients stored as YAML in `storage/statamic-mcp/oauth/clients/{client_id}.yaml`
- Client ID generated with `mcp_` prefix + 32 random hex chars
- Redirect URI validation: must be `https://` (allow `http://localhost` for dev), no fragments, absolute URIs
- Max clients cap from config — reject registration if exceeded

**Auth code storage**:
- Codes stored as JSON in `storage/statamic-mcp/oauth/codes/{hash}.json`
- Code: 64 random bytes, stored SHA-256 hashed
- PKCE challenge stored alongside code
- Expires in `code_ttl` seconds (default 600)
- `used` flag for single-use enforcement

**exchangeCode** (the critical method):
- Hash incoming code, find the JSON file
- Use `flock()` for atomic read-modify-write (prevent double-exchange)
- Validate: not used, not expired, client_id matches, redirect_uri matches
- PKCE verify: `hash_equals(base64url_encode(hash('sha256', $codeVerifier, true)), $codeChallenge)`
- Mark as used, return `OAuthAuthCode` DTO
- Throw `OAuthException` with appropriate error code on any failure

**Prune**:
- Scan clients dir, delete those older than `client_ttl`
- Scan codes dir, delete used or expired

- [ ] **Step 1: Write tests** covering:
  - `test_register_client` — creates client, returns OAuthClient with correct fields
  - `test_register_client_validates_redirect_uris` — rejects http (non-localhost), fragments
  - `test_register_client_enforces_max_clients` — rejects when cap reached
  - `test_find_client` — finds registered client
  - `test_find_client_returns_null_for_missing` — returns null
  - `test_create_and_exchange_auth_code` — full happy path with PKCE
  - `test_exchange_code_rejects_expired` — expired code throws OAuthException(invalid_grant)
  - `test_exchange_code_rejects_used` — already used code throws OAuthException(invalid_grant)
  - `test_exchange_code_rejects_wrong_client` — wrong client_id throws OAuthException(invalid_client)
  - `test_exchange_code_rejects_wrong_redirect_uri` — mismatch throws OAuthException(invalid_request)
  - `test_exchange_code_rejects_invalid_pkce` — wrong verifier throws OAuthException(invalid_request)
  - `test_prune_removes_expired_clients_and_codes` — cleans up correctly

**Files:**
- Create: `src/Http/Controllers/OAuth/OAuthTokenController.php` (named to avoid collision with CP TokenController)

`POST /mcp/oauth/token` — accepts grant_type, code, redirect_uri, client_id, code_verifier. Calls `OAuthDriver::exchangeCode()`, then `TokenService::createToken()`. Returns access_token JSON.

Token is created with:
- `userId` from the auth code
- `name` = `"{client_name} (OAuth)"`
- `scopes` from the auth code (what user approved)
- `expiresAt` = now + `oauth.token_ttl` config

Two methods:
- `show()` (GET) — validates params (client_id, response_type, code_challenge, code_challenge_method, redirect_uri, scope), renders consent screen via Inertia with client info and scopes
- `approve()` (POST) — creates auth code via `OAuthDriver::createAuthCode()`, redirects to `redirect_uri` with code + state

The GET endpoint requires Statamic CP authentication. If not logged in, redirect to CP login with `?redirect=` back to the authorize URL.

Validate:
- `response_type` must be `code`
- `code_challenge_method` must be `S256` (reject missing or `plain`)
- `client_id` must exist
- `redirect_uri` must match one from client registration
- `scope` validated against `TokenScope::cases()`
- `resource` validated against configured MCP endpoint URL

The Vue consent screen (`OAuthConsent.vue`):
- Shows client name
- Lists scopes with checkboxes (default all checked per config)
- Approve POSTs with selected scopes
- Deny redirects with `?error=access_denied&state=...`
- Pass all OAuth params as hidden fields in the form
